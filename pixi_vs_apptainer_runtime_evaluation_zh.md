# 运行时方案评估：Pixi vs Apptainer

## 问题

这个仓库应该继续主要依赖当前这种宿主机级别的 Pixi 工作流，还是应该增加一个像 Apptainer 这样的容器镜像，让流程在不同机器之间更容易迁移和复现？

## 简短结论

是的，这个仓库值得增加 Apptainer 镜像来提升运行时可复现性。

我的建议不是“彻底用 Apptainer 取代 Pixi”，而是：

- 保留 `pixi.toml` 作为开发期和依赖解析的清单
- 增加一个 Apptainer recipe，作为面向用户和可复现实验的标准运行时
- 不再把已提交的 `.pixi/` 环境目录当作主要的可移植方案

对这个项目来说，混合模式是更合适的技术选择。

## 为什么会提出这个问题

在这次真实调试中，这个仓库暴露了以下环境问题：

- `pixi` 确实安装了，但在非交互 shell 里不在 `PATH` 上
- 仓库里提交的 `.pixi/` 环境本身并不能保证可直接运行
- 这个 vendored 环境里缺少共享库链接，例如 `libcblas.so.3`
- vendored 环境里的 `Rscript` 还残留旧安装路径的硬编码前缀
- 某些二进制名称和脚本假设不一致，例如 `iqtree3` 与 `iqtree`
- PHYLIP 的辅助可执行文件位于 `.pixi/.../share/phylip-*/exe/` 这样的非直观位置
- MAD 需要手工 vendoring 才能保证定根步骤可复现

这些都不是算法问题，而是部署问题。

## 当前 Pixi 模式做得好的地方

Pixi 仍然有明显价值：

- 它是一个不错的声明式依赖清单
- 它比手写环境安装脚本更容易维护
- 在单台已知 Linux 机器上，它很适合做迭代开发
- 它能提供可锁定的软件栈，而不需要在 bash 脚本里手工列出所有依赖

如果只是一个开发者在一台机器上工作，Pixi 是够用的。

## 当前模式做得不好的地方

现在这套方案在可移植性上是偏弱的。

### 1. 已提交的 `.pixi/` 目录不是可靠制品

实际情况已经证明，复制或 vendored 的 `.pixi/` 环境很容易因为以下原因失效：

- 绝对路径假设
- wrapper 脚本按旧前缀编译或生成
- 符号链接链断裂
- 不同机器上的动态库查找差异

这意味着仓库现在背了一个很大的环境目录，但并没有真正获得稳定的可移植性。

### 2. 运行成功依赖 shell 细节

运行时是否成功，取决于很多外部细节：

- `pixi` 是否在 `PATH` 中
- `LD_LIBRARY_PATH` 是否设置
- 命令是不是以 `python3.13` 而不是 `python` 运行

这些在调试时还能接受，但并不适合作为面向用户的默认体验。

### 3. 多语言生信栈正是容器适合介入的场景

这个项目同时涉及：

- Python
- R
- PHYLIP
- IQ-TREE
- MrBayes
- MAFFT
- 一个 vendored 的 MAD 二进制

一旦从纯 Python 进入“Python + 原生生信工具 + R”的组合，宿主机级别的可复现性就会明显变差。这正是容器运行时开始体现价值的典型场景。

## 为什么 Apptainer 适合这个仓库

Apptainer 很适合这个项目，原因有几个。

### 1. 这个仓库本来就是 Linux-only

仓库已经明确偏向 Linux，这消除了 Apptainer 的一个主要障碍。

### 2. 生信用户经常运行在 HPC 上

Apptainer 在共享 Linux 服务器和 HPC 集群上很常见，而这些地方往往不允许 Docker。对系统发育工作流来说，这是很现实的优势。

### 3. 它冻结的是运行时，而不只是依赖列表

Apptainer 镜像可以一起固定：

- 精确的二进制名称
- 共享库
- wrapper 脚本
- 文件系统布局
- vendored MAD 的位置

这正是当前仓库缺少的复现层级。

### 4. 它能显著减少机器间漂移

有了 Apptainer，命令面可以变成这样：

```bash
apptainer exec egps-onebuilder.sif zsh phylotree_builder_v0.0.1/s2_phylo_4prot.zsh ...
```

这比要求每台机器都手工复原一个“刚好正确”的 Pixi shell 和动态库环境，稳定得多。

## 为什么 Apptainer 不应该完全取代 Pixi

我不建议把 Pixi 扔掉。

原因：

- Pixi 仍然是最好维护的依赖事实来源
- 开发者需要一个方便添加和更新依赖的机制
- 容器内调试通常更慢、更绕
- 容器 recipe 本身并不是一个很友好的日常依赖管理格式

所以更合理的架构是：

- Pixi 负责环境定义和本地开发
- Apptainer 负责运行时分发和可复现执行

## 建议

### 推荐方向

采用混合部署模式。

#### 开发模式

- 保留 `pixi.toml`
- 保留当前 wrapper 脚本作为本地开发入口
- 不再依赖已提交的 `.pixi/` 目录作为主要可移植机制

#### 运行模式

- 增加一个 Apptainer 定义文件
- 构建一个包含已验证工具链的统一运行镜像
- 在文档中把 Apptainer 写成用户和可重复测试的优先运行方式

### 建议强度

强烈建议增加 Apptainer 作为运行时制品。

当前环境问题的历史已经足够说明：这个项目需要从“只有依赖清单”升级到“可复现运行镜像”。

## 预期收益

如果增加 Apptainer，你可以预期：

- 更少的机器相关启动失败
- 更少的缺失库问题
- 不再依赖用户的 shell 启动文件
- 不再依赖宿主机的 `PATH` 布局
- 更少花在安装漂移上的调试时间
- 更清晰的 HPC 和服务器部署路径

## 预期成本

当然也会增加一些成本：

- 需要维护 Apptainer recipe
- 依赖变动时需要重建镜像
- 发布制品会比纯源码仓库更大
- 对想直接在宿主机看工具行为的开发者来说会增加一点摩擦

这些成本是真实存在的，但对这个仓库来说，它们小于反复修复运行时环境的成本。

## 风险与注意事项

### 1. Apptainer 不能修复工作流逻辑 bug

它解决的是打包和复现问题，不是流程内部逻辑 bug。

### 2. 大示例数据不要打进镜像

镜像应该包含软件和小体积必须资源，而不是大示例输出。

### 3. Java tanglegram 需要单独考虑

Java 模块在这个仓库里不是独立程序，不应该阻塞核心系统发育流程的容器化。

### 4. 构建宿主机要统一

因为这个仓库本质上是 Linux-only，Apptainer 的构建和测试路径也应该统一在 Linux 上。

## 实际迁移路径

### 第 1 阶段：整理运行时契约

- 保留当前 wrapper 脚本作为公开入口
- 确保两个 wrapper 都不依赖交互式 shell 状态
- 清掉主运行路径里残留的仓库外硬编码路径

### 第 2 阶段：增加 Apptainer recipe

- 在仓库根目录或 `container/` 下增加 Apptainer 定义文件
- 在镜像里安装全部运行时工具
- 把 MAD 放到与当前 wrapper 和主管线一致的位置
- 让镜像继续运行当前已经验证过的 wrapper 脚本

### 第 3 阶段：在镜像里验证一个蛋白质 demo 和一个 DNA demo

- 蛋白质 demo 应该产出方法目录和 `tree_summary/`
- DNA demo 也应产出相同结构
- 树距离统计也应该作为 smoke test 的一部分验证

### 第 4 阶段：更新 README

- 开发者保留本地 Pixi 说明
- Apptainer 作为推荐的可复现运行方式写入 README
- 明确区分“开发环境”与“运行流程”

## 最终建议

如果目标是：

- 本地开发方便：保留 Pixi
- 跨机器运行可复现：增加 Apptainer

如果只能给用户保留一种主运行机制，那么应优先选择 Apptainer，而不是已提交的 `.pixi/` 环境目录。

对这个仓库来说，这会是更稳健的运维选择。
